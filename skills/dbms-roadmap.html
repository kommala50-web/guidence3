1. Introduction to DBMS

Description: Database Management Systems (DBMS) are software systems designed to store, manage, and retrieve data efficiently. They provide mechanisms for data integrity, security, concurrency, and backup. Types include relational (RDBMS), NoSQL, hierarchical, and network databases.

// Example: Creating a table in SQL
CREATE TABLE Students (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
);

2. ER Model & Design

Description: Entity-Relationship (ER) modeling is a conceptual design tool to represent data and relationships. ER diagrams consist of entities, attributes, and relationships. Translating ER models to relational schemas is a key step in database design.

-- Student table
CREATE TABLE Student (
  student_id INT PRIMARY KEY,
  name VARCHAR(50)
);
-- Course table
CREATE TABLE Course (
  course_id INT PRIMARY KEY,
  title VARCHAR(50)
);
-- Many-to-Many: Student-Course
CREATE TABLE Enrollment (
  student_id INT,
  course_id INT,
  PRIMARY KEY(student_id, course_id),
  FOREIGN KEY(student_id) REFERENCES Student(student_id),
  FOREIGN KEY(course_id) REFERENCES Course(course_id)
);

3. Normalization

Description: Normalization organizes tables to reduce redundancy and improve data integrity. Normal forms: 1NF, 2NF, 3NF, BCNF, 4NF, 5NF. Each form addresses a specific type of anomaly (insertion, deletion, update).

4. SQL Queries

Description: SQL is the language to interact with relational databases. Basic operations: SELECT, INSERT, UPDATE, DELETE. Advanced: JOINs, GROUP BY, HAVING, subqueries, and transactions.

// Select example
SELECT name, age FROM Students WHERE age > 20;

-- Join example
SELECT S.name, C.title
FROM Student S
JOIN Enrollment E ON S.student_id = E.student_id
JOIN Course C ON E.course_id = C.course_id;

5. Transactions & ACID

Description: Transactions are sequences of operations executed as a single unit. ACID properties ensure data consistency: Atomicity, Consistency, Isolation, Durability.

// SQL Transaction example
BEGIN TRANSACTION;
UPDATE Accounts SET balance = balance - 100 WHERE id = 1;
UPDATE Accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

6. Indexing & Performance

Description: Indexes improve query performance by allowing faster searches at the cost of additional storage and write overhead. Types: single-column, composite, unique, clustered, non-clustered.

// Creating index
CREATE INDEX idx_student_name ON Student(name);

7. Views, Stored Procedures & Triggers

Description: Views provide virtual tables. Stored procedures are reusable SQL blocks. Triggers execute automatically on certain table events.

// Example: View
CREATE VIEW AdultStudents AS
SELECT * FROM Students WHERE age >= 18;

-- Trigger
CREATE TRIGGER update_log AFTER UPDATE ON Students
FOR EACH ROW
INSERT INTO StudentLog(student_id, change_time) VALUES (NEW.id, NOW());

8. Concurrency Control

Description: Concurrency control ensures that multiple transactions execute concurrently without violating data consistency. Without it, issues like lost updates, dirty reads, and uncommitted data could occur. Techniques include:

  • Pessimistic Locking (locks rows or tables to prevent conflicts)
  • Optimistic Concurrency Control (detects conflicts at commit)
  • Timestamp ordering

Question: How do you prevent two transactions from updating the same bank account balance simultaneously?

-- Lock a row for update in SQL
SELECT * FROM Accounts WHERE id = 1 FOR UPDATE;

9. Deadlocks & Resolution

Description: A deadlock occurs when two or more transactions wait indefinitely for resources held by each other. Detection methods: wait-for graph or timeout. Resolution includes rolling back one transaction or preempting resources.

Question: How would you handle a deadlock between two users trying to update overlapping records?

-- Set deadlock priority and handle in SQL
SET DEADLOCK_PRIORITY LOW;
BEGIN TRANSACTION;
UPDATE Accounts SET balance = balance - 100 WHERE id = 1;
UPDATE Accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

10. Recovery & Backup

Description: Recovery ensures database consistency after crashes or failures. Strategies include:

  • Logging: UNDO/REDO logs to revert/redo operations
  • Checkpoints: saving consistent database state
  • Backups: Full, differential, incremental

Question: How do you backup and restore a MySQL database?

-- Full backup in MySQL
mysqldump -u root -p dbname > backup.sql

-- Restore the database
mysql -u root -p dbname < backup.sql

11. Locking Mechanisms

Description: Locking is a key technique in concurrency control to prevent conflicts. Types of locks:

  • Shared Lock (S-lock): Multiple transactions can read, but cannot write.
  • Exclusive Lock (X-lock): Only one transaction can read/write a resource.
  • Intent Locks: Indicate planned future locks at a higher level (table, page).

Question: How do shared and exclusive locks help maintain data consistency?

-- Acquire a shared lock
SELECT * FROM Accounts WHERE id = 1 LOCK IN SHARE MODE;

-- Acquire an exclusive lock
SELECT * FROM Accounts WHERE id = 1 FOR UPDATE;

12. Two-Phase Locking (2PL)

Description: Two-Phase Locking ensures serializability of transactions in DBMS. It has two phases:

  • Growing Phase: Locks are acquired, no locks released.
  • Shrinking Phase: Locks are released, no new locks acquired.

Question: How does 2PL prevent conflicts and maintain consistency?

-- Example in SQL pseudo-code
BEGIN TRANSACTION;
-- Growing phase: acquire locks
SELECT * FROM Accounts WHERE id = 1 FOR UPDATE;
SELECT * FROM Accounts WHERE id = 2 FOR UPDATE;

-- Perform updates
UPDATE Accounts SET balance = balance - 100 WHERE id = 1;
UPDATE Accounts SET balance = balance + 100 WHERE id = 2;

-- Shrinking phase: release locks at commit
COMMIT;

13. Isolation Levels

Description: Isolation levels control how transactions interact with each other. They help prevent phenomena like dirty reads, non-repeatable reads, and phantom reads. Common levels:

  • Read Uncommitted
  • Read Committed
  • Repeatable Read
  • Serializable

Question: Which isolation level prevents dirty reads but allows non-repeatable reads?

-- Set transaction isolation level
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

14. Replication

Description: Replication duplicates data across multiple servers to improve availability, fault tolerance, and read performance. Types include:

  • Master-Slave Replication
  • Master-Master Replication
  • Synchronous vs Asynchronous Replication

Question: How would you configure replication in MySQL to ensure high availability?

-- Master configuration
CHANGE MASTER TO
  MASTER_HOST='master_host',
  MASTER_USER='replica_user',
  MASTER_PASSWORD='password',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=107;
START SLAVE;

16. Indexing Strategies

Description: Indexing optimizes query performance. Common strategies include:

  • Single-column vs Composite Index
  • Unique Indexes
  • Full-text Index
  • Covering Index
  • Clustered vs Non-clustered

Question: How do covering indexes improve query performance?

-- Composite index example
CREATE INDEX idx_student_name_age ON Student(name, age);

-- Full-text index example
CREATE FULLTEXT INDEX idx_course_title ON Course(title);

17. Materialized Views

Description: Materialized Views store query results physically to improve performance for complex queries. Unlike normal views, they do not calculate data on the fly but require refresh mechanisms.

Question: How would you use a materialized view to optimize reporting queries?

-- Create a materialized view in Oracle
CREATE MATERIALIZED VIEW StudentCourseSummary
REFRESH FAST
AS
SELECT S.student_id, COUNT(E.course_id) AS total_courses
FROM Student S
JOIN Enrollment E ON S.student_id = E.student_id
GROUP BY S.student_id;

18. Partitioning

Description: Partitioning splits large tables into smaller, manageable pieces to improve query performance and maintenance. Types include:

  • Range Partitioning
  • List Partitioning
  • Hash Partitioning
  • Composite Partitioning

Question: How would you partition a sales table by year to optimize queries?

-- Range Partitioning example in Oracle
CREATE TABLE Sales (
  sale_id INT,
  sale_date DATE,
  amount NUMBER
)
PARTITION BY RANGE (sale_date) (
  PARTITION p_2023 VALUES LESS THAN (TO_DATE('01-JAN-2024','DD-MON-YYYY')),
  PARTITION p_2024 VALUES LESS THAN (TO_DATE('01-JAN-2025','DD-MON-YYYY'))
);

19. NoSQL Basics

Description: NoSQL databases are non-relational systems designed for scalability and flexible data models. Types include document, key-value, column-family, and graph databases.

Question: How would you store user profiles with variable fields using NoSQL?

// MongoDB example: storing user profiles as documents
db.users.insertOne({

20. Big Data Integration

Description: Modern DBMS often integrate with Big Data systems for analytics. Techniques include ETL pipelines, Hadoop/Spark integration, and streaming data ingestion.

Question: How would you integrate MySQL with Apache Spark for analytics?

// Spark SQL example in PySpark
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("MySQLIntegration").getOrCreate()
df = spark.read.format("jdbc").options(

  dbtable="Sales",

df.show()

21. MongoDB Advanced Operations

Description: MongoDB supports aggregation, indexing, and replication for advanced use-cases. Aggregation pipelines allow filtering, grouping, and transforming data efficiently.

Question: How would you calculate the total orders per customer in MongoDB?

// Aggregation pipeline example
db.orders.aggregate([
  { $group: { _id: "$customer_id", totalOrders: { $sum: "$amount" } } },
  { $sort: { totalOrders: -1 } }
]);
